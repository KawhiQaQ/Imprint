## 核心概念

在深入算法细节之前，先统一定义本文档中反复出现的核心概念：

### POI 与候选池

**POI**（Point of Interest）是地图上的一个兴趣点，在本系统中指景点、餐厅或酒店。系统从高德地图 API 搜索大量 POI 后，经过筛选存入**候选池**——一个待选的 POI 集合。后续所有的排程、优化都是从候选池中挑选 POI 来填充行程。

### 旅行角色（Tourism Role）

每个景点在行程中扮演的"角色"，决定了它会被安排在一天中的什么时段、占多长时间。系统定义了 8 种角色：

| 角色 | 含义 | 示例 |
|------|------|------|
| CORE_ATTRACTION | 城市名片，必去的核心景点 | 西湖、故宫 |
| MAJOR_AREA | 可逛 2-4 小时的大型区域 | 古镇、大型景区 |
| NATURE_RELAX | 自然休闲 | 湖、绿道、森林公园 |
| NIGHT_EXPERIENCE | 夜间体验 | 夜市、灯光秀 |
| CULTURAL_SITE | 文化场馆 | 博物馆、美术馆 |
| VIEWPOINT | 打卡拍照点 | 观景台、地标建筑 |
| SHOPPING_AREA | 商圈/步行街 | 步行街、市集 |
| FILLER | 顺路看看的小景点 | 小祠堂、普通广场 |

角色的核心作用：上午黄金时段优先安排 CORE_ATTRACTION，晚间安排 NIGHT_EXPERIENCE，FILLER 只在没有更好选择时才会被选上，且是被替换的首选目标。

### 体验域（Experience Domain）

比景点类型更高一层的语义分类，代表一种"城市体验"。爬山和看瀑布是不同景点，但都属于 NATURAL_LANDSCAPE（自然风光）这个体验域。系统定义了 9 种体验域：

NATURAL_LANDSCAPE（自然风光）、CULTURAL_LANDMARK（文化地标）、HISTORIC_TOWN（古镇老街）、WATER_SCENERY（水景）、URBAN_LIFE（城市生活）、ENTERTAINMENT（娱乐休闲）、RELIGIOUS_SITE（宗教场所）、MUSEUM_ART（博物馆/美术馆）、PARK_GARDEN（公园/园林）。

体验域的核心作用：确保行程覆盖了这座城市最值得体验的几个方面，而不是 5 天全在逛公园。

### 景点类别（Category）

比体验域更细一层的分类，描述景点的具体物理形态。一个体验域下面会包含多个景点类别，例如 NATURAL_LANDSCAPE（自然风光）这个体验域下有 mountain（山）、lake（湖）、river（河）、forest（森林）、scenic_area（风景区）等多个类别。系统定义了 20+ 种类别：

mountain（山）、lake（湖）、river（河）、sea（海）、forest（森林）、scenic_area（风景区）、ancient_town（古镇）、temple（寺庙）、historic_building（历史建筑）、ancestral_hall（宗祠）、memorial（纪念馆）、museum（博物馆）、art_gallery（美术馆）、garden（园林）、park（公园）、botanical（植物园）、zoo（动物园）、commercial_street（商业街）、plaza（广场）等。

景点类别的核心作用：在每日主题规划和体验区域排名中，用于更精确地匹配"今天想逛什么类型的地方"。比如同样是"自然山水日"，category 能区分出今天偏山还是偏湖。

### 锚点（Anchor）

从候选池中选出的、**必须出现在最终行程中的景点**。锚点是硬约束——不管后续怎么优化调整，锚点都不能被删掉。每个"必须体验"的体验域至少有一个锚点。锚点还有一个时间权重：大景点（如山岳）权重 1.0 独占一整天，中景点（如祖庙）权重 0.5 占半天。

### 聚类与区域

**聚类**是将地理位置相近的景点归成一组的过程，每组称为一个**区域（Cluster）**。一个区域大致对应"一天可以逛完的地理范围"。在区域内部，距离 1.5km 以内的景点会进一步聚合成**体验区域（Experience Area）**——一个可以步行游览的连片小片区。

### 角色位（Role Slot）

每天的时间表被切成固定的几个格子，每个格子对应一个时段（如上午景点、午餐、下午景点），并且对景点的旅行角色有明确偏好。系统往格子里填景点的过程称为"角色位填充"。

## 整体架构

行程规划管道（Pipeline）分为七个阶段，依次执行：

```
用户输入（目的地、天数、偏好）
    |
    v
+-------------------------------------+
|  第一阶段：POI 多通道召回            |  <- 高德地图 API
|  5 个并行通道，~500 POI 原始召回     |
|  配额采样 -> 距离去偏 -> ~200 POI    |
+-----------------+-------------------+
                  |
                  v
+-------------------------------------+
|  第二阶段：POI 语义标注              |  <- LLM (DeepSeek)
|  旅行角色分类（8 类）                |
|  体验域分类（9 类）                  |
+-----------------+-------------------+
                  |
                  v
+-------------------------------------+
|  第三阶段：旅行时间图聚类            |  <- Union-Find 算法
|  构建可达性图 -> 连通分量发现         |
|  合并/拆分 -> 天数分配               |
+-----------------+-------------------+
                  |
                  v
+-------------------------------------+
|  第四阶段：城市体验覆盖模型          |  <- LLM + 约束求解
|  城市画像 -> 锚点选择 -> 域配额       |
|  硬约束：锚点必须出现在行程中        |
+-----------------+-------------------+
                  |
                  v
+-------------------------------------+
|  第五阶段：体验区域识别与日程编排     |  <- 凝聚聚类 + 角色位填充
|  步行区域聚合 -> 主题匹配 -> 时段分配 |
+-----------------+-------------------+
                  |
                  v
+-------------------------------------+
|  第六阶段：路径优化                  |  <- 贪心近邻 + 2-opt
|  时段内路径优化 -> 折返检测修复       |
+-----------------+-------------------+
                  |
                  v
+-------------------------------------+
|  第七阶段：质量保障与自修复          |  <- LLM 审查 + 算法修复
|  锚点验证 -> 域配额验证 -> 去重       |
|  餐厅/景点合理性审查 -> 自动替换     |
+-------------------------------------+
```

核心设计原则：
- **LLM 不做排序**：LLM 擅长语义理解（"这是什么类型的景点"），但不擅长空间推理（"这两个景点距离多远"）。所有涉及距离、顺序、时间的决策都由算法完成。
- **硬约束优先**：锚点覆盖和体验域配额是硬约束，不满足 = 行程不合法，必须修复。
- **多级回退**：每个阶段都有 fallback 机制，确保在数据不足时仍能生成合理行程。

## 第一阶段：POI 多通道召回

> 源码位置：`backend/src/services/poiService.ts` -> `searchAttractionsByPreference()`

### 问题背景

高德地图 API 的搜索结果存在**城市中心密度偏置**：市中心的小景点（广场、街心花园）数量远多于郊区的大景点（山、森林公园），直接搜索会导致候选池被低质量 POI 淹没。

### 多通道召回架构

系统采用 5 个并行召回通道，各自独立搜索，最后按配额混合填入候选池：

| 通道 | 名称 | 策略 | 配额 |
|------|------|------|------|
| NATURAL | 自然景观专用 | 14 个固定关键词（山、森林公园、风景区等），每词 3 页 | 25 |
| LANDMARK | 城市地标 | "城市名+后缀"组合查询（如"佛山景区"、"佛山公园"），每词 2 页 | 25 |
| USER | 用户兴趣 | 基于用户偏好的活动类型搜索，最多 5 个活动 | 20 |
| TYPECODE | 类别枚举 | 按高德 typecode 定向扫描 9 个类型（山岳、湖泊、古镇等），每类 2 页 | 20 |
| AROUND | 周边扩展 | 以城市中心为圆心，50km 半径 searchAround | 20 |

### 配额采样（Quota Sampling）

各通道先各自保留 TopN（按评分排序），再按比例混合，候选池总量上限 250 个 POI。

排序策略：有评分的优先 -> 评分高的优先 -> 有 typecode 的优先（说明是正规景点）。

### 距离去偏（Geo Debias）

按距城市中心的距离分桶，每桶有保底数量和上限，确保郊区大景点不被市中心小景点淹没：

| 距离桶 | 保底数量 | 上限 |
|--------|---------|------|
| 0-5km | 10 | 60 |
| 5-15km | 8 | 50 |
| 15-40km | 6 | 40 |
| 40km+ | 4 | 30 |

### 质量检查与自修复

对关键体验域（自然景观、文化地标、水景、古镇）进行检查：如果候选池中某个域没有评分 >= 3.5 的高质量 POI，自动触发扩展召回（更多关键词 + 更大半径 + searchAround），让系统自我修复。

### 低质量过滤

通过关键词黑名单过滤明显不适合旅游的 POI：社区、居委会、停车场、医院、银行、超市、小区等。

> **本阶段的作用**：从高德地图的海量数据中，构建一个数量充足（~200 个）、地理分布均衡、质量有保障的候选池。这个候选池是后续所有阶段的数据基础——后面的标注、聚类、排程都是在这个池子里挑选和组合。

## 第二阶段：POI 语义标注

> 源码位置：`backend/src/services/poiService.ts` -> `classifyTourismRoles()` / `classifyExperienceDomains()`

候选池中的 POI 目前只有名称、地址、评分等基础信息，系统还不知道每个景点的"角色"和"体验类型"。这一阶段为每个景点赋予前文定义的两层语义标签：旅行角色和体验域。

### 分类流程

1. **规则预分类**：基于景点名称和高德 typecode 进行快速规则匹配（如名称含"山"-> `mountain` 类别，含"博物馆"-> `museum` 类别），作为 LLM 的 fallback。
2. **LLM 精确分类**：将候选池中所有景点列表发送给 DeepSeek，由 LLM 根据景点名称、类别、地址、评分等信息进行精确的旅行角色和体验域分类。
3. **双保险**：如果 LLM 调用失败，规则预分类的结果仍然可用。

### 名称归一化与去重

系统会对景点名称进行归一化处理，防止候选池中同一景点的不同入口/分区被当作不同景点：
- 去掉括号内容：`广州塔(E区)` -> `广州塔`
- 去掉方位后缀：`景区东门` -> `景区`
- 互相包含检测：`广州塔` 和 `广州塔E区` 视为同一景点

> **本阶段的作用**：将候选池中"只有名称和坐标"的原始 POI，升级为"带有旅行角色和体验域标签"的语义化 POI。这两层标签是后续所有阶段的决策依据——聚类需要知道景点的重要程度（角色），体验覆盖模型需要知道景点属于哪种城市体验（体验域），角色位填充需要按角色匹配时段。没有这一步，后面的阶段就只能按距离和评分做决策，无法理解景点的语义含义。

## 第三阶段：旅行时间图聚类

> 源码位置：`backend/src/services/clusterService.ts`

### 核心思想

传统的 KMeans 聚类按直线距离分组，但旅行规划关心的是"两个景点能否在同一天游览"。系统使用 **Travel-Time Graph Clustering**：两个景点属于同一天，不是看直线距离，而是看从 A 出发访问 B 再回酒店是否合理。聚类的结果是若干个**区域（Cluster）**，每个区域对应"一天可以逛完的地理范围"。

### 算法步骤

#### Step 1：构建可达性图 + Union-Find

以候选池中的景点为主节点，构建无向图：

```
对于每对景点 (A, B)：
    distance = 直线距离(A, B)
    if distance <= 15km:    // 15km 直线 ~ 20-25km 实际路程 ~ 40min 车程
        添加边 (A, B)
```

使用 **Union-Find（并查集）** 数据结构发现连通分量，每个连通分量就是一个区域。

Union-Find 实现了路径压缩和按秩合并优化：
- `find(x)`：路径压缩，将节点直接指向根
- `union(x, y)`：按秩合并，矮树挂到高树上
- `getGroups()`：返回所有连通分量

#### Step 2：餐厅和酒店分配

景点形成主聚类后，将候选池中的餐厅和酒店分配到最近的区域：
- 餐厅：距区域质心 <= 18km（阈值 x 1.2）
- 酒店：距区域质心 <= 22.5km（阈值 x 1.5，酒店可以稍远）

#### Step 3：小区域合并

如果区域数量超过天数，贪心合并最小的区域到最近的大区域：
- 按景点数量排序，大区域优先
- 每次合并最小区域到最近的邻居
- 但如果最近邻距离 > 30km（2x 阈值），说明是独立远郊区域（如西樵山），保持独立

#### Step 4：大区域拆分

如果只有 1 个区域但行程 >= 4 天，沿最长轴（经度或纬度）均分拆分：
- 4-6 天拆 2 个子区域
- 7-9 天拆 3 个子区域
- 拆分后重新分配餐厅和酒店

#### Step 5：天数分配

按景点数量比例分配天数，基准为每天 3.5 个景点：

```
每个区域的建议天数 = max(1, min(3, ceil(景点数 / 3.5)))
```

约束：
- 每个区域至少 1 天
- 单个区域最多 3 天
- 按地理顺序排列（贪心：从最大区域出发，每次选最近的未访问区域），减少跨区域移动

> 在这里就会输出一个列表，比如5天行程就输出[zone-1, zone-1, zone-2, zone-2, zone-3]，在这里就决定后续角色位填充时，第一天应该聚焦于哪个区域簇。（不过这个列表第四阶段还可能会修改）

### 距离计算

使用经纬度近似公式计算欧氏距离：

```
dLng = (lng1 - lng2) x 111 x cos(lat x pi / 180)    // 经度差转千米
dLat = (lat1 - lat2) x 111                           // 纬度差转千米
distance = sqrt(dLng^2 + dLat^2)
```

> **本阶段的作用**：将之前散落在候选池中的 ~200 个独立景点，按实际旅行可达性划分为若干个地理区域，每个区域对应"一天可以逛完的范围"。这一步从"一堆点"变成了"几块区域 + 天数分配"，为后续的日程编排提供了空间骨架——后面的体验覆盖、角色位填充、路径优化都是在区域内部进行的，不会出现一天之内跨越整座城市的不合理安排。

## 第四阶段：城市体验覆盖模型

> 源码位置：`backend/src/services/itineraryPipeline.ts` -> `generateOptimizedItinerary()` 前半部分

这是行程规划的**硬约束层**。核心理念：来一座城市旅行，有些体验是"不体验就白来了"的，这些体验必须出现在行程中。本阶段利用前面定义的体验域和锚点概念，建立行程的底线约束。

### 4.1 城市画像生成

由 LLM 分析目的地城市，输出两个列表：
- `mustHave`：必须体验的体验域（通常 2-4 个），如佛山 -> `[CULTURAL_LANDMARK, NATURAL_LANDSCAPE, HISTORIC_TOWN]`
- `optional`：可选体验的体验域，如 `[URBAN_LIFE, PARK_GARDEN]`

### 4.2 锚点选择（Anchor Selection）

从候选池中为每个体验域选出锚点。选择流程分四轮：

1. **mustHave 域锚点**：每个 mustHave 体验域从候选池中选评分最高的景点作为锚点
2. **optional 域锚点**：如果天数充裕（锚点总时间权重 < 天数），为 optional 体验域也选锚点
3. **第二锚点**：为热门体验域选第二个高分景点，防止遗漏
4. **全局高分兜底**：不限体验域，选候选池中剩余的高分非 FILLER 景点

#### 候选池反向校验

如果候选池中某个体验域有 CORE_ATTRACTION 角色的景点，但该域不在城市画像中，自动补入 optional 列表。这防止了 LLM 生成的城市画像遗漏重要体验域。

### 4.3 体验域配额（Domain Quota）

硬约束——不是加权，是约束。定义每个 mustHave 体验域在行程中至少要占多少天：

```
5 天行程示例：
  自然景观 >= 1 天
  文化地标 >= 1 天
  城市生活 >= 1 天
```

不满足配额 = 行程不合法，触发自动修复（用该体验域的高分景点替换 FILLER）。

### 4.4 酒店策略

根据行程长度决定酒店策略：
- **短行程（<= 3 天）**：单酒店，选距景点中心最近的连锁酒店
- **长行程（>= 4 天）**：多酒店策略，跨区域时换酒店，每个区域选一家

### 4.5 锚点分配策略

就是简单的排队，比如选出几个锚点后，分全天锚点（timeWeight >= 1.0，如山、古镇），再分半天锚点（timeWeight = 0.5，如祖庙、博物馆），按顺序占天：
1. 全天锚点：按选出的顺序，依次占 Day 1、Day 2、Day 3...每个独占一天
2. 半天锚点配对：按评分从高到低，两两尝试配对——如果两个半天锚点距离 < 18km，就共享同一天（半天 + 半天 = 一整天）
3. 未配对的半天锚点：各自独占一天

然后根据分配的锚点情况，进行zone的重新修改，比如：
```
锚点 A：西樵山（zone-3，全天，timeWeight=1.0）→ 占 Day 1
锚点 B：祖庙（zone-1，半天，timeWeight=0.5）  ┐
锚点 C：梁园（zone-1，半天，timeWeight=0.5）  ┘ 距离 < 18km，配对 → 共享 Day 2

原始分配：[zone-1, zone-1, zone-2, zone-2, zone-3]
锚点分配后：Day 1 的 zone-1 改成 zone-3（因为西樵山在 zone-3）
最终分配：[zone-3, zone-1, zone-2, zone-2, zone-3]
```

> 实际上并没有人工把锚点强制塞进某一天。在 5.3 构建当天候选列表时，锚点的 score 会被直接覆写为 10.0（跳过正常的评分公式，正常景点的 score 通常在 0.3 ~ 1.2 之间）。这个 10.0 在后续角色位填充中作为 combinedScore 公式的 baseScore 参与计算，分数碾压其他候选，锚点自然会被选中。此外，全天锚点（timeWeight=1.0）还会将当天的候选范围限制在锚点 10km 内，让那天的行程自然围绕锚点展开。

> **本阶段的作用**：在聚类划分好的地理区域之上，叠加一层"体验约束"——确定哪些城市体验是必须覆盖的，并为每种必须体验选出锚点景点。这一步从"区域里有哪些景点"变成了"哪些景点必须出现、每种体验至少安排几天"，为行程建立了质量底线。后续的日程编排和质量保障都以这里的锚点和配额作为硬约束，保证最终行程不会遗漏城市最核心的体验。

## 第五阶段：体验区域识别与日程编排

> 源码位置：`backend/src/services/itineraryPipeline.ts` -> `identifyExperienceAreas()` / `optimizeRoute()` / `generateDayFramework()`

### 5.1 体验区域识别

一个聚类区域可能有 30 个 POI，但实际上分布在 3 个可步行游览的小片区。系统使用**凝聚聚类**将区域内的 POI 进一步聚合成体验区域（Experience Area）——每个体验区域是一个半径 <= 1.5km 的步行可达片区：

```
算法：
1. 按评分降序排列区域内所有景点
2. 从最高分景点开始，作为体验区域核心（seed）
3. 将距 seed <= 1.5km 的未分配景点加入该体验区域
4. 重复直到所有景点都被分配
5. 同时将距体验区域核心 <= 1.5km 的餐厅分配到该区域
```

每个体验区域计算**质量分**：

```
qualityScore = maxScore x 0.5 + avgScore x 0.3 + roleQuality x 0.2
```

其中 roleQuality 基于区域内最高旅行角色等级（CORE_ATTRACTION: 1.0, MAJOR_AREA: 0.8, NATURE_RELAX: 0.7, CULTURAL_SITE: 0.6, FILLER: 0.1）。

> 后续角色位填充时，是先给当天所有体验区域按主题匹配度排名，把排名靠前的体验区域里的景点收集出来，组成当天的候选列表。角色位填充时，从这个候选列表里逐个挑景点填入每个时段

### 5.2 每日主题规划

LLM 为每天生成主题（如"岭南文化日"、"自然山水日"），每个主题包含：
- 偏好的旅行角色列表（按优先级排序）
- 偏好的景点类别关键词

### 5.3 体验区域排名与分数传导

为每天选择体验区域时，按主题匹配度排名：

```
themeScore = qualityScore + themeBonus

themeBonus 计算：
  + 0.40  区域内最多的景点角色，恰好是今天主题想要的角色（如"自然山水日"偏好 NATURE_RELAX，而这个区域大部分景点就是 NATURE_RELAX）
  + 0.25  区域内最多的景点类别（category，如 mountain/lake/temple 等具体形态分类，比体验域更细），恰好是今天主题想要的类别（如主题偏好 mountain/lake，而这个区域主要就是山和湖）
  + 0.20  区域内有城市名片级景点（CORE_ATTRACTION）或大型景区（MAJOR_AREA）
  - 0.30  区域内超过一半的景点是低质量的 FILLER（说明这个区域整体质量不高）
```

体验区域的排名不只是决定"先从哪个区域收集景点"，还会直接影响景点的分数。收集景点进入当天候选列表时，每个景点的分数会被放大：

```
lScore = baseScore × (1 + 所在体验区域的 themeScore × 0.15)
```

举例：themeScore = 1.70 的体验区域，里面的景点分数会乘以 1.255（+25.5%）；themeScore = 0.05 的体验区域，景点分数几乎不变（×1.007）。这样在后续角色位填充时，排名靠前的体验区域里的景点天然占优势，更容易被选中。

### 5.4 每日时间框架与角色位

系统为每天生成固定的时段框架。每个景点时段就是一个角色位（Role Slot），对景点的旅行角色有明确偏好：

| 时段 | 时间 | 类型 | 偏好的旅行角色 | 角色权重 |
|------|------|------|--------------|---------|
| breakfast | 08:00 | 餐厅 | - | - |
| morning | 09:30 | 景点 | CORE, MAJOR, NATURE | 2.5 |
| lunch | 12:00 | 餐厅 | - | - |
| afternoon | 14:00 | 景点 | MAJOR, NATURE, CORE, CULTURAL | 1.5 |
| dinner | 18:00 | 餐厅 | - | - |
| evening | 20:00 | 景点 | NIGHT, SHOPPING, VIEWPOINT | 0.5 |

特殊处理：
- **第一天**：根据到达时间裁剪时段（如下午到达则跳过早餐和上午）
- **最后一天**：根据离开时间裁剪时段（如下午离开则跳过晚餐和晚间）
- **大景点日**：如果当天有时间权重 >= 1.0 的锚点，候选范围限制在锚点 10km 内

### 5.5 角色位填充（Role Slot Filling）

这是日程编排的核心算法。不再让候选池中所有 POI 在综合分上平等竞争，而是为每个角色位定向筛选：

```
对于每个角色位：
    1. 构建当前角色位的候选子集：
       - 景点角色位：从体验区域中收集旅行角色匹配的 POI
       - 餐厅角色位：优先体验区域内餐厅，不足则扩大搜索范围
       - 早餐角色位：直接搜索附近早餐店（不从通用餐厅候选池选）
    
    2. 计算综合评分：
       combinedScore = lScore x roleWeight x themeBonus - distancePenalty - categoryPenalty
       
       其中：
       - lScore：基于体验区域排名放大后的 POI 基础评分（见 5.3 和 5.6），注意这里的 baseScore 已经在 5.3 构建候选列表时被体验区域的 themeScore 系数放大过了，不是最原始的评分
       - roleWeight：当前时段偏好的旅行角色与景点角色是否匹配（比如上午时段偏好 CORE_ATTRACTION，如果景点恰好是 CORE_ATTRACTION 则权重 1.0，不匹配则降为 0.3-0.7）
       - themeBonus：景点是否符合今天的主题方向（比如今天是"自然山水日"，自然类景点会获得额外加分）
       - distancePenalty：距离惩罚（见 5.7）
       - categoryPenalty：类别重复惩罚（同类别景点越多惩罚越大）
    
    3. 选择最高分的候选
    4. 更新当前位置、已用 POI 集合、类别计数
```

### 5.6 POI 评分公式

```
baseScore = baseRating + categoryBonus + roleBonus + ratingBonus

baseRating = rating / 5（无评分默认 0.5）

categoryBonus（景点类别加减分）：
  山/湖/海/森林/风景区:  +0.20 ~ +0.25
  古镇/历史建筑/园林:    +0.15
  寺庙/博物馆:           +0.10
  普通公园:              -0.05
  广场:                  -0.10
  大学:                  -0.30
  宗祠:                  -0.20

roleBonus（旅行角色加减分）：
  CORE_ATTRACTION:  +0.20
  MAJOR_AREA:       +0.15
  NATURE_RELAX:     +0.10
  FILLER:           -0.15

ratingBonus：
  评分 >= 4.5:  +0.15
  评分 >= 4.0:  +0.05
```

### 5.7 距离惩罚

餐厅和景点使用不同的距离惩罚参数：

| 参数 | 餐厅 | 景点 |
|------|------|------|
| 理想距离 | <= 2km | <= 5km |
| 可接受距离 | <= 5km | <= 10km |
| 硬上限 | <= 8km | <= 15km |
| 理想区惩罚/km | 0.2 | 0.1 |
| 可接受区惩罚/km | 0.4 | 0.2 |
| 硬上限区惩罚/km | 0.8 | 0.4 |

### 5.8 候选池多级回退

当体验区域内的候选不足时，按以下顺序逐步扩大搜索范围：

```
体验区域内 POI
    | 不足 4 个景点
当前聚类区域内所有 POI
    | 仍不足
邻接聚类区域（质心距离 <= 18km）内 POI
    | 仍不足
15km 半径内 POI
    | 仍不足
25km 半径内 POI
    | 仍不足
邻接聚类区域全部 POI（兜底，不跨全城）
```

**注意：**

1. 总的来说就是，先计算体验区域得分。把所有体验区域景点放进一个“当天候选池”中。然后基于上述各个公式计算每个景点的baseScore，再根据体验区域得分计算lScore。然后基于其它几个变量（用于景点在哪个时间段的规划）计算combinedScore作为景点综合评分。注意，如果某个景点是锚点，那么无论它计算出来的baseScore是多少，都覆写为10（远高于正常计算的baseScore），保证锚点的选择。最后会根据combinedScore作为当天每个时段的选择依据。

2. 候选池不是按评分排好序等着被挑的。角色位填充是逐个时段进行的，每个时段从候选池里算一遍 combinedScore 然后选最高分的。因为 distancePenalty 是基于"当前位置"算的，而当前位置会随着每选一个景点而更新，所以同一个景点在不同时段的 combinedScore 是不一样的。不是一次性排好序，而是每个时段重新算一次。

### 5.9 完整示例：一天的日程是怎么编排出来的

以佛山 5 天行程的第 2 天为例，假设当天主题是"岭南文化日"，所在区域是禅城区，区域内有 3 个体验区域：

**体验区域排名**

| 体验区域 | qualityScore | themeBonus | 最终 themeScore |
|----------|-------------|------------|----------------|
| 祖庙周边（主导角色 CORE_ATTRACTION，主导类别 temple） | 0.85 | +0.40（角色匹配）+0.25（类别匹配）+0.20（有 CORE） = +0.85 | 1.70 |
| 梁园周边（主导角色 CULTURAL_SITE，主导类别 garden） | 0.70 | +0.20（有 MAJOR_AREA）= +0.20 | 0.90 |
| 某广场周边（主导角色 FILLER，主导类别 plaza） | 0.35 | -0.30（FILLER 占比 > 50%）= -0.30 | 0.05 |

系统按 themeScore 排名，优先从"祖庙周边"收集候选景点，其次"梁园周边"，"某广场周边"排最后。

**角色位填充过程**

当天的时段框架：早餐 → 上午景点 → 午餐 → 下午景点 → 晚餐 → 晚间景点。逐个时段填充：

| 时段 | 偏好角色 | 候选与评分计算 | 选中 |
|------|---------|--------------|------|
| 上午景点 | CORE_ATTRACTION | 祖庙：baseScore 0.90 × roleWeight 1.0（角色匹配）× themeBonus 1.15 - 距离惩罚 0.05 = **0.98**；某公园：baseScore 0.50 × roleWeight 0.4（角色不匹配）× themeBonus 1.0 - 距离惩罚 0.02 = **0.18** | 祖庙 ✓ |
| 下午景点 | MAJOR_AREA, CULTURAL_SITE | 梁园：baseScore 0.75 × roleWeight 1.0 × themeBonus 1.10 - 距离惩罚 0.10 = **0.73**；岭南天地：baseScore 0.65 × roleWeight 0.7 × themeBonus 1.05 - 距离惩罚 0.03 - categoryPenalty 0.10（已有 temple 类）= **0.35** | 梁园 ✓ |
| 晚间景点 | NIGHT_EXPERIENCE | 岭南天地：baseScore 0.65 × roleWeight 0.8 × themeBonus 1.0 - 距离惩罚 0.08 = **0.44** | 岭南天地 ✓ |

注意：每选中一个景点后，当前位置会更新为该景点的位置，后续候选的距离惩罚都基于新位置计算。同时已选景点不会再出现在候选中，同类别景点的 categoryPenalty 也会累加。

> **本阶段的作用**：将前面阶段产出的"区域 + 锚点 + 配额"转化为具体的每日行程安排——每天去哪几个景点、吃哪几家餐厅、按什么顺序。这一步是整个管道中信息密度最高的阶段：体验区域识别把大区域细分为步行可达的小片区，主题规划赋予每天独特的游览主题，角色位填充按时段特性精准匹配景点。经过这一阶段，行程从抽象的"区域和约束"变成了可执行的"时间表"。

## 第六阶段：路径优化

> 源码位置：`backend/src/services/routeOptimizer.ts`

### 典型场景说明

在当前的角色位框架下，每个时段（上午、下午、晚间）通常只安排 1 个景点。因此本阶段的贪心近邻和 2-opt 算法在典型场景下不会触发（节点数 < 2 时直接跳过）。真正在典型场景下发挥作用的是**折返检测**——它跨时段检查整天的节点序列，发现并修复回头路。

贪心近邻和 2-opt 作为能力储备，会在以下非典型场景中触发：
- 第七阶段质量修复后往某个时段插入了额外景点
- 未来扩展为单时段多景点的框架

### 折返检测与修复（典型场景下的核心逻辑）

折返检测不区分时段，而是看整天所有节点的序列。比如一天的节点是 早餐 → A（上午）→ 午餐 → B（下午）→ 晚餐 → C（晚间），它会检查任意连续三个节点是否存在折返：

```
if distance(A,B) + distance(B,C) > distance(A,C) x 1.5:
    // 折返！尝试交换 B 和 C
    if distance(A,C) + distance(C,B) < distance(A,B) + distance(B,C):
        swap(B, C)
```

### 贪心近邻 + 2-opt（非典型场景）

当某个时段内有多个景点时，系统会对该时段内的景点做路径优化：

#### 整体流程

```
1. 分离固定节点和可优化节点
   - 固定节点：arrival, breakfast, lunch, dinner, hotel, departure（保持位置不变）
   - 可优化节点：morning, afternoon, evening 角色位中的景点

2. 按时段分组优化（仅当该时段有 >= 2 个景点时才触发）
   - 上午景点：以酒店位置为起点，贪心近邻 -> 2-opt
   - 下午景点：贪心近邻 -> 2-opt
   - 晚间景点：贪心近邻 -> 2-opt -> 晚间特殊优化（最后一个离酒店最近）

3. 重建节点列表，保持固定节点位置不变
```

#### 贪心近邻算法（Greedy Nearest Neighbor）

构建初始路径：每次选择距离当前位置最近的未访问节点。

```
function greedyNearestNeighbor(nodes, startLocation):
    result = []
    remaining = copy(nodes)
    current = startLocation
    
    while remaining is not empty:
        nearest = argmin(distance(current, node) for node in remaining)
        result.append(nearest)
        remaining.remove(nearest)
        current = nearest.location
    
    return result
```

时间复杂度：O(n^2)，其中 n 为节点数（通常 <= 5）。

#### 2-opt 局部搜索

在贪心近邻的基础上，通过反转路径段来减少总距离：

```
function twoOpt(nodes):
    best = nodes
    bestDistance = totalDistance(best)
    improved = true
    iterations = 0
    
    while improved and iterations < 50:
        improved = false
        iterations++
        
        for i in 0..n-2:
            for j in i+2..n-1:
                // 尝试反转 [i+1, j] 段
                newRoute = best[0..i] + reverse(best[i+1..j]) + best[j+1..n]
                newDistance = totalDistance(newRoute)
                
                if newDistance < bestDistance:
                    best = newRoute
                    bestDistance = newDistance
                    improved = true
    
    return best
```

时间复杂度：O(n^2 x iterations)，其中 n <= 5，iterations <= 50。

#### 晚间特殊优化

晚间活动的最后一个节点应该离酒店最近，方便回酒店休息：按到酒店距离降序排列晚间节点，距离最远的排最前，距离最近的排最后。

#### 距离缓存

使用 Map 缓存已计算的距离，避免重复调用高德 API。双向缓存：同时存储 A->B 和 B->A。

> **本阶段的作用**：对上一阶段生成的每日时间表进行路径层面的精细优化——在不改变景点选择的前提下，调整同一时段内景点的访问顺序，消除折返和绕路。这一步将"选对了景点但顺序随意"的行程，优化为"顺序合理、路线流畅"的行程，直接影响用户实际出行的交通时间和体验感。

## 第七阶段：质量保障与自修复

> 源码位置：`backend/src/services/itineraryPipeline.ts` -> 后半部分

行程生成后，系统执行多轮质量检查和自动修复，确保最终行程的合理性。

### 7.1 锚点覆盖验证（硬约束）

检查所有锚点是否出现在最终行程中。缺失的锚点触发强制插入：

```
修复策略：
1. 找到锚点被分配的那一天
2. 在该天的景点节点中，按替换优先级排序：
   - FILLER 角色 -> 优先替换
   - 低分非锚点 -> 其次
   - 其他锚点 -> 不可替换
3. 优先替换 morning 角色位中的 FILLER
4. 如果没有可替换节点，直接插入
```

### 7.2 体验域配额验证（硬约束）

检查每个 mustHave 体验域是否满足最低天数配额：

```
修复策略：
1. 从候选池中找到缺失体验域的最佳未使用景点
2. 找到没有该体验域覆盖的天（优先选 FILLER 最多的天）
3. 用该体验域景点替换 FILLER 或最低分景点
```

### 7.3 去重

#### 景点去重
- 名称完全相同 -> 重复
- 名称互相包含（如"广州塔"和"广州塔E区"）-> 重复
- 归一化后相同 -> 重复
- 距离 < 500m -> 重复

替换策略：从候选池找替代，距被替换节点 1.5km 内的候选跳过（大概率是同一景区的子设施）。候选池耗尽时，用高德实时搜索附近 10km 内的景点。

#### 餐厅去重
- 名称完全相同 -> 重复
- 从候选池找未使用的替代

### 7.4 LLM 审查（三轮）

#### 第一轮：餐厅合理性审查

LLM 检查所有餐厅安排：
- 同类型餐厅过多（连续多天都是包子/粥/面馆）
- 早餐安排了正餐类餐厅（酒楼、火锅、海鲜）
- 午餐/晚餐安排了非正餐类店铺（甜品店、奶茶店）
- 整体缺乏多样性

不合理的餐厅通过高德 searchAround 搜索附近 5km 内的替代餐厅。

#### 第二轮：无效景点审查

LLM 识别明显不合理的景点（最多替换 3 处）：公司/企业、学校/教育机构、社区/居民区、政府/行政机构、名称过于模糊泛化的地方。

#### 第三轮：景点多样性审查

LLM 审查景点安排的多样性（最多替换 5 处）：宗祠过多、同质化严重（如连续多个类似的公园）、缺乏体验层次感。

### 7.5 框架完整性检查

最终兜底检查：确保每天的角色位框架完整（不缺少早餐、午餐、晚餐等必要时段）。前面的审查和替换步骤可能间接导致时段缺失，这里做最后一次修复。

### 7.6 LLM 描述生成

为所有景点和餐厅节点生成自然的推荐描述（20-40 字），像本地人推荐一样：
- 景点：描述特色亮点、推荐游玩内容
- 餐厅：描述特色菜品、推荐点什么
- 早餐店：推荐当地特色早点

> **本阶段的作用**：作为整个管道的最后一道防线，对已生成的完整行程进行多维度的质量审查和自动修复。锚点验证确保核心景点没有丢失，配额验证确保体验多样性达标，去重确保没有重复景点，LLM 审查捕捉算法难以发现的语义问题（如餐厅类型不合理、景点同质化）。经过这一阶段，行程从"算法生成的初稿"变成了"可以直接交付给用户的成品"。

## 使用案例

假设用户输入"佛山，5 天，喜欢自然风光和历史文化"，以下是算法从头到尾的推进过程：

- **第一阶段（POI 多通道召回）**：系统同时启动 5 个搜索通道向高德地图 API 发起请求。NATURAL 通道用"山"、"森林公园"、"风景区"等 14 个关键词搜索自然景观；LANDMARK 通道用"佛山景区"、"佛山公园"等组合词搜索城市地标；USER 通道根据用户偏好搜索对应活动类型；TYPECODE 通道按高德类型编码定向扫描山岳、湖泊、古镇等 9 个类型；AROUND 通道以佛山市中心为圆心做 50km 半径的周边搜索。5 个通道共召回约 500 个原始 POI。然后进行配额采样，每个通道按配额保留 TopN，混合后上限 250 个。接着距离去偏，按距市中心的距离分桶（0-5km、5-15km、15-40km、40km+），每桶有保底数量，防止市中心小景点淹没郊区大景点。最后过滤掉停车场、医院、超市等明显不适合旅游的 POI，并做质量检查，缺少高质量 POI 的关键词自动触发扩展召回。最终得到约 200 个质量有保障、地理分布均衡的候选池。
&nbsp;
- **第二阶段（POI 语义标注）**：候选池中的 200 个 POI 目前只有名称、地址、评分，系统还不知道它们各自是什么"角色"。先用规则做一轮预分类（名称含"山"归为 mountain，含"博物馆"归为 museum），作为兜底，这里是分出类别。然后将整个候选池发送给 LLM，由 LLM 为每个景点标注旅行角色（8 类，如 CORE_ATTRACTION、FILLER）和体验域（9 类，如 NATURAL_LANDSCAPE、CULTURAL_LANDMARK）。同时做名称归一化去重，把"广州塔"和"广州塔E区"识别为同一景点。经过这一步，每个 POI 从"一个坐标点"变成了"带有语义标签的景点"。
&nbsp;
- **第三阶段（旅行时间图聚类）**：用 Union-Find 算法将候选池中的景点按地理可达性分组。对每对景点计算直线距离，距离 <= 15km 的连一条边，然后找连通分量，每个连通分量就是一个区域（zone）。接着把餐厅和酒店分配到最近的区域。如果区域太多（超过天数），贪心合并最小的区域到最近的邻居（但距离 > 30km 的独立远郊区域保持独立）。如果只有 1 个区域但天数 >= 4 天，沿最长轴拆分成 2-3 个子区域。最后按景点数量比例分配天数，输出一个列表如 `[zone-1, zone-1, zone-2, zone-2, zone-3]`，决定每天应该聚焦于哪个区域。
&nbsp;
- **第四阶段（城市体验覆盖模型）**：LLM 分析佛山这座城市，输出 mustHave 体验域（如文化地标、自然景观、古镇老街）和 optional 体验域（如城市生活、公园园林）。然后从候选池中为每个 mustHave 域选出评分最高的景点作为锚点（如祖庙代表文化地标，西樵山代表自然景观），天数充裕时也为 optional 域选锚点。每个锚点有时间权重：西樵山是大景点（1.0，独占一天），祖庙是中景点（0.5，占半天）。锚点按排队顺序分配到具体的天，全天锚点先占天，半天锚点两两配对（距离 < 18km 的共享一天）。分配完后，如果锚点所在的 zone 和上一阶段分配的那天原本的 zone 不一致，就修改那天的 zone 分配，让区域服从锚点。同时生成体验域配额（如自然景观 >= 1.5 天、文化地标 >= 1 天），作为后续质量检查的硬约束。
&nbsp;
- **第五阶段（体验区域识别与日程编排）**：这是信息密度最高的阶段。首先在每个大区域内部，用 1.5km 半径做二次聚类，把景点聚合成步行可达的体验区域（如"祖庙周边"、"梁园周边"）。然后 LLM 为每天生成主题（如"岭南文化日"、"自然山水日"），每个主题带有偏好的旅行角色和景点类别。每个体验区域的景点先计算baseScore（锚点的 basescore 被覆写为 10.0）。接着按体验区域得分公式计算分数，给当天的体验区域排名，排名靠前的区域里的景点的基础评分baseScore获得更大分数加成，加成后的分数为lScore。所有体验区域的景点汇入当天候选列表。最后逐个时段做角色位填充：每个时段从候选列表里计算所有景点的 combinedScore（= lScore × roleWeight × themeBonus - distancePenalty - categoryPenalty），选最高分的填入。每选一个景点就更新当前位置，下一时段的分配将重新计算所有候选的距离惩罚。一天填完后进入下一天，直到所有天都编排完毕。
&nbsp;
- **第六阶段（路径优化）**：在当前框架下每个时段只有 1 个景点，所以贪心近邻和 2-opt 不会触发。真正发挥作用的是折返检测：跨时段检查整天的节点序列，如果发现连续三个节点存在回头路（AB + BC > AC × 1.5），就尝试交换后两个节点的顺序。这一步不增减景点，只调整顺序，消除明显的绕路。
&nbsp;
- **第七阶段（质量保障与自修复）**：对完整行程做多轮检查和修复。第一轮验证所有锚点是否出现在行程中，缺失的强制插入（替换 FILLER）。第二轮验证体验域配额是否满足（如自然景观是否 >= 1.5 天），不满足就从候选池找该域景点替换进去。第三轮做景点和餐厅去重（名称相同、距离 < 500m 等判定为重复）。然后 LLM 做三轮审查：审查餐厅合理性（早餐不该是火锅、不该连续多天吃同类型）、审查无效景点（公司、学校、社区等不该出现）、审查景点多样性（宗祠过多、同质化严重）。不合理的通过高德实时搜索附近替代。最后做框架完整性兜底检查，确保每天不缺少必要时段。全部修复完成后，LLM 为每个景点和餐厅生成 20-40 字的推荐描述，行程交付给用户。

## 关键数据结构

### TravelNode（行程节点）

```typescript
interface TravelNode {
  id: string;
  name: string;
  type: 'attraction' | 'restaurant' | 'hotel' | 'transport';
  address: string;
  description: string;
  dayIndex: number;          // 第几天
  order: number;             // 当天内的顺序
  timeSlot: string;          // 角色位：arrival/breakfast/morning/lunch/afternoon/dinner/evening/hotel/departure
  scheduledTime: string;     // 计划时间，如 "09:30"
  estimatedDuration: number; // 预计时长（分钟）
  location: string;          // 经纬度 "lng,lat"
  transportMode: string;     // 交通方式：walk/bus/subway/taxi
  transportDuration: number; // 交通时长（分钟）
}
```

### POICluster（聚类区域）

```typescript
interface POICluster {
  id: string;                    // 如 "zone-1"
  name: string;                  // 如 "禅城区区域"
  centroid: { lng, lat };        // 区域质心
  attractions: EnrichedPOI[];    // 区域内景点
  restaurants: EnrichedPOI[];    // 区域内餐厅
  hotels: EnrichedPOI[];         // 区域内酒店
  avgInternalDistance: number;   // 内部平均距离（km）
  radius: number;                // 区域半径（km）
}
```

### AnchorPOI（锚点）

```typescript
interface AnchorPOI {
  poi: EnrichedPOI;
  domain: ExperienceDomain;  // 所属体验域
  score: number;             // 评分
  assignedDay?: number;      // 分配到的天
  timeWeight?: number;       // 时间权重：1.0=整天，0.5=半天
}
```

### ExperienceArea（体验区域）

```typescript
interface ExperienceArea {
  id: string;
  name: string;                        // 如 "祖庙周边"
  centroid: { lng, lat };
  radius: number;                      // 半径（km）
  attractions: ScoredPOI[];
  restaurants: ScoredPOI[];
  dominantRole: TourismRole;           // 主导旅行角色
  dominantCategory: string;            // 主导类别
  qualityScore: number;                // 质量分
}
```

## 算法复杂度与性能

| 阶段 | 算法 | 时间复杂度 | 典型规模 |
|------|------|-----------|---------|
| POI 召回 | 5 通道并行搜索 | O(通道数 x 页数) | ~500 原始 POI |
| 配额采样 | 排序 + 截断 | O(n log n) | 500 -> 250 |
| 距离去偏 | 分桶 + 排序 | O(n log n) | 250 -> ~200 |
| 语义标注 | LLM 批量调用 | O(1) per LLM call | 2 次 LLM 调用 |
| 旅行时间图聚类 | Union-Find | O(n^2) | n <= 200 景点 |
| 小区域合并 | 贪心 | O(k^2) | k <= 10 区域 |
| 大区域拆分 | 排序 + 均分 | O(n log n) | n <= 200 |
| 体验区域识别 | 凝聚聚类 | O(n^2) | n <= 50 per cluster |
| 角色位填充 | 贪心选择 | O(n x slots) | n <= 50, slots <= 6 |
| 路径优化（贪心近邻） | 贪心 | O(n^2) | n <= 5 per segment |
| 路径优化（2-opt） | 局部搜索 | O(n^2 x iter) | n <= 5, iter <= 50 |
| 质量保障 | LLM 审查 + 搜索 | O(issues) | 3-5 次 LLM 调用 |

整体 LLM 调用次数：每次行程生成约 10-15 次 LLM 调用（分类、主题规划、质量审查、描述生成等）。

## 约束与阈值汇总

| 约束 | 值 | 说明 |
|------|-----|------|
| 旅行时间阈值 | 15km | 两景点直线距离 <= 15km 视为可同日游览 |
| 餐厅分配阈值 | 18km | 餐厅距区域质心 <= 18km |
| 酒店分配阈值 | 22.5km | 酒店距区域质心 <= 22.5km |
| 独立区域阈值 | 30km | 最近邻 > 30km 的区域保持独立 |
| 体验区域半径 | 1.5km | 步行可达范围 |
| 邻接聚类距离 | 18km | 质心距离 <= 18km 视为邻接 |
| 每天景点基准 | 3.5 个 | 用于计算区域天数分配 |
| 单区域最大天数 | 3 天 | 超过 3 天必须换区 |
| POI 评分下限 | 0.3 | 低于 0.3 的景点被过滤 |
| 景点去重距离 | 500m | 距离 < 500m 视为同一景点 |
| 替换排斥距离 | 1.5km | 替换候选距被替换节点 < 1.5km 跳过 |
| 2-opt 最大迭代 | 50 次 | 路径优化最大迭代次数 |
| 折返惩罚系数 | 1.5 | AB+BC > AC x 1.5 判定为折返 |
| 候选池上限 | 250 个 | 配额采样后的最大 POI 数 |
